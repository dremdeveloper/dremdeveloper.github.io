# instruction_following_eval(google-research)

> 이 저장소는 모델이 “똑똑한가”를 묻는 도구가 아닙니다.  
> **모델이 사람이 준 지시를 끝까지, 정확한 형식으로, 기계적으로 검증 가능한 수준에서 지켰는가**를 묻는 도구입니다.  
> 화려하진 않습니다. 대신 정직합니다. 이런 평가는 원래 그래야 합니다.

작성 기준: 2026-03-24  
대상 저장소: `google-research/google-research/tree/master/instruction_following_eval`

---

## 1. 먼저 한 줄로 요약하겠습니다

이 디렉터리는 **IFEval(Instruction Following Eval)** 구현체입니다.  
핵심 아이디어는 단순합니다.

- “400단어 이상으로 써라”
- “쉼표를 쓰지 마라”
- “응답 전체를 JSON으로 감싸라”
- “4번째 문단을 `elm`으로 시작하라”

이런 식의 **검증 가능한(verifiable) 지시**를 미리 정의해 두고,  
모델 응답이 그 조건을 만족하는지 **규칙 기반으로 자동 채점**합니다.

즉, 이 저장소는 “답변 품질”보다 **지시 준수율**을 보는 데 특화되어 있습니다.

---

## 2. 이 저장소가 왜 의미가 있나

LLM 평가는 대개 두 가지가 뒤섞입니다.

1. 내용이 좋은가
2. 시킨 대로 했는가

현업에서는 의외로 2번이 더 치명적일 때가 많습니다.  
좋은 말을 길게 했는데 JSON으로 달라니까 마크다운 문단으로 뱉고,  
쉼표 넣지 말랬는데 습관적으로 쉼표를 넣고,  
두 개 답을 달랬는데 한 개만 주면 파이프라인은 바로 깨집니다.

IFEval은 바로 그 지점을 찌릅니다.

- 사람 평가자를 붙이지 않아도 되고
- 다른 LLM을 심판으로 세우지 않아도 되고
- 규칙만 맞으면 누구나 같은 결과를 재현할 수 있습니다

연구용으로도 좋고,  
실무에서는 **“우리 모델이 지시를 얼마나 잘 지키는가”**를 확인하는 회귀 테스트 세트로 써먹기 좋습니다.

---

## 3. 이 저장소의 큰 그림

이 코드는 생각보다 단순합니다. 흐름은 이렇습니다.

```text
입력 프롬프트 집합(input_data.jsonl)
            +
모델 응답 파일(input_response_data_*.jsonl)
            ↓
evaluation_main.py
            ↓
evaluation_lib.py
            ↓
instruction_id별 checker class 실행
            ↓
strict / loose 두 방식으로 채점
            ↓
jsonl 결과 파일 저장 + 콘솔 리포트 출력
```

여기서 중요한 건 **“instruction_id별 checker class 실행”**입니다.  
실제 채점은 거의 전부 `instructions.py` 안의 클래스들이 담당합니다.

## 4. 디렉터리 구조를 30년차 개발자 시선으로 읽어보자

### `data/`
벤치마크 입력과 샘플 응답이 들어 있습니다.

- `input_data.jsonl`  
  평가용 프롬프트 원본입니다.
- `input_response_data_gpt4_20231107_145030.jsonl`  
  같은 프롬프트들에 대한 GPT-4 응답 예시입니다.

이 저장소는 데이터가 따로 숨겨져 있지 않습니다.  
즉, “코드만 공개”가 아니라 **코드와 샘플 데이터가 같이** 들어 있습니다.  
이게 꽤 중요합니다. 재현성의 절반은 데이터 배포 방식에서 결정되니까요.

### `README.md`
의존성 설치와 기본 실행 예시가 있습니다.  
설명은 짧습니다. 아주 짧습니다.  
그래서 이 문서를 따로 쓰는 겁니다.

### `evaluation_main.py`
CLI 엔트리 포인트입니다.  
입력 파일을 읽고, strict / loose 평가를 각각 돌리고, 결과 jsonl을 저장하고, 리포트를 출력합니다.

### `evaluation_lib.py`
실제 평가 파이프라인의 본체입니다.

- 입력 JSONL 로딩
- 응답 JSONL 로딩
- strict 평가
- loose 평가
- 출력 JSONL 저장
- 통계 리포트 출력

즉, `main`이 껍데기라면, 여기가 몸통입니다.

### `instructions.py`
이 저장소의 심장입니다.  
각 지시 유형마다 checker class가 하나씩 들어 있습니다.

예를 들면 이런 것들입니다.

- 특정 키워드 포함 여부
- 특정 키워드 금지
- 단어 수 / 문장 수 제약
- JSON 형식 여부
- 쉼표 금지
- 큰따옴표로 감싸기
- 두 개의 응답을 `******`로 분리하기
- 4번째 문단 첫 단어 강제하기

실무자 관점에서 보면 이 파일이 거의 제품입니다.

### `instructions_registry.py`
instruction id를 checker class에 매핑하는 파일입니다.

즉,

- `keywords:frequency` → `KeywordFrequencyChecker`
- `detectable_format:json_format` → `JsonFormat`
- `punctuation:no_comma` → `CommaChecker`

이런 식으로 연결합니다.

또한 **충돌 규칙(conflict table)** 도 같이 들고 있습니다.  
예를 들어 JSON 강제와 특정 키워드 금지는 조합이 애매해질 수 있고,  
대문자 강제와 소문자 강제를 같이 넣는 건 당연히 모순입니다.

### `instructions_util.py`
보조 유틸입니다.

- 언어 코드 매핑
- 문장 분리
- 단어 수 계산
- 문장 수 계산
- 랜덤 키워드 생성

눈에 띄는 점은 NLTK에 꽤 기대고 있다는 겁니다.  
겉보기에는 소박하지만, 문장 수/단어 수 기준을 제대로 세려면 이런 유틸이 꼭 필요합니다.

### `*_test.py`
테스트 코드입니다.  
이 저장소는 “규칙 기반 평가기”라서 테스트가 특히 중요합니다.  
이런 류의 코드는 멋보다 **경계 조건 테스트**가 생명입니다.

### `requirements.txt`
의존성은 비교적 작습니다.

- `absl`
- `langdetect`
- `nltk`
- `immutabledict`

### `run.sh`
예제 실행용 래퍼입니다.  
복잡한 스크립트는 아닙니다. “이렇게 돌리면 된다”는 최소 예시라고 보면 됩니다.

## 5. 입력 데이터는 어떻게 생겼나

이 저장소는 JSONL을 씁니다.  
즉, **한 줄에 한 개의 JSON 객체**입니다.

### 5-1. `input_data.jsonl` 구조

대략 이런 모양입니다.

```json
{
  "key": 2250,
  "prompt": "사용자에게 보여줄 원본 프롬프트",
  "instruction_id_list": ["length_constraints:nth_paragraph_first_word"],
  "kwargs": [
    {
      "first_word": "elm",
      "num_paragraphs": 4,
      "nth_paragraph": 4
    }
  ]
}
```

여기서 핵심은 세 가지입니다.

- `prompt`: 실제 모델에게 준 프롬프트
- `instruction_id_list`: 어떤 규칙들로 채점할지
- `kwargs`: 그 규칙에 들어갈 파라미터

즉, 프롬프트 텍스트 자체만 있는 게 아니라  
**“이 프롬프트는 어떤 규칙으로 채점할 것인가”**가 같이 들어 있습니다.

### 5-2. `input_response_data_*.jsonl` 구조

응답 파일은 더 단순합니다.

```json
{
  "prompt": "input_data.jsonl에 있던 prompt와 정확히 같은 문자열",
  "response": "모델이 생성한 응답"
}
```

여기서 아주 중요한 실무 포인트 하나.

> 이 코드는 `prompt` 문자열 자체를 키로 써서 응답을 매칭합니다.

즉, 공백 하나 다르거나 줄바꿈 하나 달라도 매칭이 안 됩니다.  
이런 방식은 단순해서 빠르지만, **문자열 정확 일치에 아주 민감**합니다.

## 6. 실제 실행 흐름을 코드 기준으로 풀어보자

### 6-1. `evaluation_main.py`

이 파일은 세 개의 플래그를 받습니다.

- `input_data`
- `input_response_data`
- `output_dir`

그리고 다음 순서로 움직입니다.

1. 입력 프롬프트 목록을 읽습니다.
2. 프롬프트→응답 딕셔너리를 만듭니다.
3. `strict`, `loose` 두 평가 함수를 각각 돌립니다.
4. 결과를 `eval_results_strict.jsonl`, `eval_results_loose.jsonl`로 저장합니다.
5. 콘솔에 정확도 리포트를 출력합니다.

이 파일의 미덕은 **욕심이 없다는 것**입니다.  
평가 로직을 억지로 여기 넣지 않고, 조용히 orchestration만 합니다.

### 6-2. `evaluation_lib.py`

여기서 진짜 일이 벌어집니다.

#### `InputExample`, `OutputExample`
`dataclass`로 입력/출력 포맷을 정리해 둡니다.  
작지만 좋은 습관입니다. 이런 파일 포맷 중심 코드에서는 특히 그렇습니다.

#### `read_prompt_list`
`input_data.jsonl`을 읽어 `InputExample` 목록으로 바꿉니다.

#### `read_prompt_to_response_dict`
응답 파일을 읽어 `{prompt: response}` 딕셔너리를 만듭니다.

#### `test_instruction_following_strict`
말 그대로 엄격 채점입니다.

- 응답 원문을 그대로 사용
- instruction id마다 checker class 생성
- `check_following(response)` 호출
- 전부 만족하면 `follow_all_instructions=True`

이 방식은 **있는 그대로** 봅니다.  
모델이 앞뒤에 군더더기를 붙였든, 마크다운을 써서 형식을 흐렸든, 그 책임을 그대로 묻습니다.

#### `test_instruction_following_loose`
느슨한 채점입니다.  
여기가 꽤 흥미롭습니다.

이 함수는 응답 하나만 검사하지 않고, 여러 변형본을 같이 검사합니다.

- 원문
- `*` 제거본
- 첫 줄 제거본
- 마지막 줄 제거본
- 첫 줄+마지막 줄 제거본
- 그리고 각 변형본에서 `*`까지 제거한 버전

즉, **모델이 실제로는 지시를 대체로 지켰는데, 래퍼 한 줄이나 마크다운 강조 때문에 실패하는 경우**를 좀 더 너그럽게 보는 방식입니다.

개인적으로 이 loose 모드는 이름을 잘 지었다고 봅니다.  
“정답 판정”이 아니라, **상한선(upper bound) 추정**에 가깝습니다.

#### `print_report`
리포트는 세 겹으로 나옵니다.

1. `prompt-level`  
   한 프롬프트에 붙은 지시들을 전부 지켰는가
2. `instruction-level`  
   전체 지시 개수 중 몇 개를 지켰는가
3. 카테고리별 정확도  
   - 대분류(`tier0`): 예를 들어 `keywords`, `language`, `detectable_format`
   - 세부 규칙(`tier1`): 예를 들어 `keywords:frequency`, `startend:quotation`

이 구성이 실용적입니다.  
PM은 prompt-level을 보고,  
연구자는 instruction-level과 세부 규칙 분포를 봅니다.

## 7. strict와 loose를 어떻게 이해해야 하나

이 둘을 헷갈리면 안 됩니다.

### strict
“실전 배포 기준”에 가깝습니다.  
모델이 실제로 낸 출력 그대로 검사합니다.

이 모드가 중요한 이유는 간단합니다.  
프로덕션 시스템은 `의도는 좋았는데 포맷이 살짝 어긋남`을 봐주지 않기 때문입니다.

### loose
“모델이 대충은 알아들었는가”를 보는 완화 버전입니다.  
특히 다음 같은 상황을 구제합니다.

- 맨 앞에 쓸데없는 한 줄이 붙음
- 맨 뒤에 덧붙임 멘트가 붙음
- 강조용 `*` 때문에 형식 검사가 깨짐

즉, strict는 배포 친화적이고, loose는 진단 친화적입니다.

## 8. `instructions.py`는 어떻게 읽어야 하나

이 파일은 체크 규칙들을 모아 둔 클래스 라이브러리입니다.  
구조는 거의 모두 비슷합니다.

- `build_description(...)`
- `get_instruction_args()`
- `get_instruction_args_keys()`
- `check_following(value)`

이 패턴이 중요한 이유는,  
**규칙의 설명 생성**과 **규칙의 검증 로직**을 같은 클래스 안에 묶어 두었기 때문입니다.

연구 코드답게 솔직합니다.  
“이 규칙이 뭔지”와 “그 규칙을 어떻게 검사하는지”를 한데 붙여 둡니다.  
읽는 사람 입장에서는 아주 편합니다.

## 9. 어떤 지시 유형들이 있나

현재 registry 기준으로 **25개 instruction type**이 등록되어 있습니다.

제가 실무자 관점으로 묶어서 설명하면 아래와 같습니다.

### 9-1. 키워드/문자 관련
- 특정 키워드 포함
- 특정 키워드 빈도
- 특정 단어 금지
- 특정 글자 빈도

이건 마케팅 카피, 템플릿 생성, 규칙형 응답에서 자주 필요합니다.

### 9-2. 언어/대소문자 관련
- 응답 전체를 특정 언어로 작성
- 영어 대문자만 사용
- 영어 소문자만 사용
- 전부 대문자인 단어의 개수 제약

이건 사람이 보기엔 사소해 보여도, 시스템 통합에서는 꽤 자주 요구됩니다.

### 9-3. 길이/구조 관련
- 문장 수 제약
- 단어 수 제약
- 문단 수 제약
- N번째 문단의 첫 단어 고정

이류 시스템 같아 보여도, 보고서 자동화나 템플릿 자동 생성에서는 매우 실용적입니다.

### 9-4. 포맷/표식 관련
- placeholder 개수
- postscript 존재 여부
- bullet list 개수
- 강조된 섹션 수
- section 개수
- JSON 형식
- 제목(`<<title>>`)
- 큰따옴표로 감싸기
- 쉼표 금지

즉, 이 저장소는 “내용 평가기”라기보다 **출력 규약 검증기**에 가깝습니다.

### 9-5. 복합 행동 관련
- 정해진 선택지 중 하나로만 답하기
- 두 개의 다른 응답을 구분자로 분리하기
- 프롬프트를 먼저 그대로 반복한 뒤 답하기
- 특정 문구로 정확히 끝내기

이 부분이 꽤 재밌습니다.  
단순 형식 검사에서 한 단계 올라가, **응답 순서와 행동 패턴**까지 건드립니다.

## 10. 몇몇 checker를 보면 성격이 더 잘 보입니다

### `JsonFormat`
응답 전체가 JSON인지 검사합니다.  
다만 코드 블록 fence, 예를 들면 ```` ```json ```` 같은 래퍼는 벗겨낸 뒤 파싱합니다.

즉, “JSON을 마크다운 코드블록 안에 담아낸 경우”도 통과시켜 줍니다.  
실무적으로 꽤 합리적입니다.

### `BulletListChecker`
정확히 몇 개의 bullet point가 있는지 봅니다.  
`*`와 `-` 둘 다 bullet로 인정합니다.

즉, 문법을 아주 빡빡하게 한 종류로만 막진 않습니다.

### `TwoResponsesChecker`
응답 두 개를 `******`로 분리했는지 봅니다.  
그리고 두 응답이 서로 달라야 통과합니다.

이런 건 간단해 보여도, 실제로는 모델들이 자주 실수하는 타입입니다.

### `RepeatPromptThenAnswer`
이 클래스는 프롬프트를 먼저 그대로 반복한 뒤 답하라는 패턴을 검사합니다.

다만 구현을 보면 체크는 꽤 단순합니다.  
**응답이 해당 프롬프트로 시작하는지만 봅니다.**  
즉, 뒤에 진짜 “답”이 이어졌는지까지는 엄밀하게 보지 않습니다.

연구 코드에서 흔히 보이는 선택입니다.  
복잡도를 낮추고, 가장 중요한 신호만 잡겠다는 거죠.

### `TitleChecker`
응답 안에 `<< ... >>` 형태의 제목이 있으면 통과시킵니다.  
이건 아주 실용적인 규칙입니다. 문서형 응답을 강제할 때 유용합니다.

### `QuotationChecker`
응답 전체가 큰따옴표(`"`)로 감싸져 있는지 봅니다.  
단순하지만, 이런 제약은 다른 모델보다 LLM이 더 자주 틀립니다.

---

## 11. 이 저장소를 읽으면서 꼭 알아둘 함정들

이제부터가 진짜 실무 메모입니다.  
겉보기 README에는 잘 안 보이는데, 코드를 보면 보이는 포인트들입니다.

### 11-1. `input_response_data`는 플래그 정의상 optional처럼 보여도, 사실상 필수입니다
플래그 선언은 `required=False`인데,  
실제 `main`에서는 이 값을 바로 읽어서 파일을 엽니다.

즉, 값을 안 주면 “없어도 된다”가 아니라 거의 바로 깨진다고 보는 게 맞습니다.

이건 연구 코드에서 흔히 나오는 사소한 불일치입니다.  
큰 문제는 아니지만, 처음 쓰는 사람은 한번쯤 넘어집니다.

### 11-2. 응답 매칭은 `prompt` 문자열 exact match입니다
이건 단순하고 빠르지만 대단히 민감합니다.

- 공백 차이
- 줄바꿈 차이
- 후처리로 인한 escape 차이

이런 게 있으면 바로 대응이 안 됩니다.

내 경험상 이런 벤치마크를 CI에 넣을 때는  
반드시 **프롬프트 직렬화 규칙을 고정**해야 합니다.

### 11-3. `loose`는 꽤 관대합니다
마크다운 `*`를 날리고, 앞줄/뒷줄도 잘라 봅니다.

그러니 loose 점수는 “모델이 이 지시를 이해했는가”를 보는 참고치로는 좋지만,  
**실전 통합 안정성 지표**로 쓰기에는 너무 부드럽습니다.

### 11-4. “문단”의 정의가 checker마다 다릅니다
이건 아주 중요합니다.

- `ParagraphChecker`는 문단 구분자를 `***`로 봅니다.
- `ParagraphFirstWordCheck`는 문단 구분자를 `\n\n`로 봅니다.

즉, 같은 “paragraph”라는 말을 쓰지만 **구현 정의가 서로 다릅니다**.

이건 버그라기보다 설계 타협에 가깝습니다.  
다만 확장할 때 반드시 주의해야 합니다.

### 11-5. 언어 감지 실패는 실패가 아니라 통과 처리됩니다
`langdetect`가 예외를 던지면 `False`가 아니라 `True`를 줍니다.

짧은 텍스트, 이상한 문자열, 혼합 표기에서는 이게 관측될 수 있습니다.  
즉, 언어 관련 점수는 아주 짧은 응답에서는 약간 낙관적일 수 있습니다.

### 11-6. 어떤 규칙은 “정확히”가 아니라 “최소한”입니다
예를 들어,

- 강조 섹션 수는 **최소 개수 이상**
- section 수 역시 **최소 개수 이상**
- placeholder 개수도 **최소 개수 이상**
- bullet 개수는 **정확히 일치**

즉, 규칙마다 동등 비교(`==`)인지, 하한 비교(`>=`)인지 다릅니다.  
새 규칙을 추가할 때 이 철학을 먼저 맞춰야 합니다.

### 11-7. registry에 비활성화된 규칙이 보입니다
코드를 보면 rephrase 관련 항목이나 multi-turn constrained start 같은 것들이  
주석 처리된 흔적으로 남아 있습니다.

이 말은 곧, 이 저장소가 “완성형 제품”이라기보다  
**연구 실험을 거쳐 정리된 코드베이스**라는 뜻입니다.

연구 코드에서 이런 흔적은 나쁜 게 아닙니다.  
오히려 어떤 규칙이 왜 제외되었는지 짐작할 수 있어 좋습니다.

---

## 12. `instructions_registry.py`는 왜 중요한가

대부분의 사람은 `instructions.py`만 보고 끝내는데,  
실은 registry도 꽤 중요합니다.

이 파일은 두 가지를 합니다.

1. **instruction id → checker class 매핑**
2. **instruction conflict 정의**

예를 들어,

- `change_case:english_capital`
- `change_case:english_lowercase`

이 둘을 같이 쓰는 건 자명하게 모순입니다.

또 JSON 형식 강제는 거의 모든 다른 규칙과 충돌 가능성이 큽니다.  
registry는 이런 상성을 표 형태로 들고 있습니다.

다만 현재 보이는 평가 경로(`evaluation_main.py` → `evaluation_lib.py`)에서는  
실제 채점 시 `INSTRUCTION_DICT`는 쓰지만, conflict table을 직접 참조하진 않습니다.

제 해석은 이렇습니다.

> conflict 정보는 **데이터 생성/조합 단계**나,  
> 향후 확장용 안전장치 성격으로 둔 흔적에 가깝습니다.

즉, 지금 당장 점수 계산에 쓰이는 중심 정보는 **매핑 테이블**입니다.

---

## 13. `instructions_util.py`는 소박하지만 꽤 중요합니다

좋은 평가기에서 제일 자주 발목 잡히는 건  
거창한 모델이 아니라 **사소한 문자열 처리**입니다.

이 유틸 파일은 그런 지점을 담당합니다.

- 문장 분리
- 단어 수 세기
- 언어 코드 관리
- 랜덤 키워드 생성

특히 문장 수/단어 수 규칙은  
사람이 셀 때와 코드가 셀 때가 자주 다릅니다.

이 저장소는 그 기준을 코드로 명시합니다.  
즉, 결과가 마음에 안 들어도 적어도 **왜 그런 결과가 나왔는지 추적 가능**합니다.  
이게 중요합니다. 재현성의 핵심이니까요.

## 14. 실제로 어떻게 돌리면 되나

기본 흐름은 이렇습니다.

### 14-1. 의존성 설치
```bash
pip3 install -r instruction_following_eval/requirements.txt
```

### 14-2. 내 모델 응답 파일 준비
예를 들어:

```json
{"prompt": "프롬프트 A", "response": "모델 응답 A"}
{"prompt": "프롬프트 B", "response": "모델 응답 B"}
```

주의할 점은 오직 하나입니다.

> `prompt` 문자열은 입력 데이터의 `prompt`와 **완전히 같아야** 합니다.

### 14-3. 평가 실행
```bash
python3 -m instruction_following_eval.evaluation_main \
  --input_data=./instruction_following_eval/data/input_data.jsonl \
  --input_response_data=./my_model_outputs.jsonl \
  --output_dir=./instruction_following_eval/data/
```

### 14-4. 결과 확인
대략 다음 산출물이 생깁니다.

- `eval_results_strict.jsonl`
- `eval_results_loose.jsonl`

그리고 콘솔에는 prompt-level / instruction-level / 카테고리별 정확도가 찍힙니다.

## 15. 결과 파일은 어떻게 읽어야 하나

출력 JSONL에는 대략 이런 정보가 들어갑니다.

```json
{
  "instruction_id_list": ["detectable_format:json_format", "punctuation:no_comma"],
  "prompt": "...",
  "response": "...",
  "follow_all_instructions": false,
  "follow_instruction_list": [true, false]
}
```

여기서 실무적으로 중요한 건 `follow_instruction_list`입니다.  
이걸 보면 **정확히 어느 규칙에서 미끄러졌는지** 알 수 있습니다.

모델을 개선할 때는 전체 점수보다 이런 실패 패턴을 보는 편이 훨씬 생산적입니다.

예를 들면,

- JSON은 잘 맞추는데 쉼표 금지에서 계속 깨진다
- 길이 제약은 잘 맞추는데 언어 감지 쪽이 불안하다
- repeat_prompt 류에서 습관적으로 앞말을 덧붙인다

이런 식의 진단이 가능해집니다.

## 16. 이 저장소의 장점

### 장점 1. 재현성이 좋다
사람 심사나 LLM judge에 덜 기대므로, 결과를 다시 뽑기 쉽습니다.

### 장점 2. 해석이 쉽다
왜 점수가 깎였는지 비교적 명확합니다.  
이건 연구보다 실무에서 더 빛납니다.

### 장점 3. 비용이 싸다
한번 모델 응답만 받아 두면 채점은 거의 문자열 처리 수준입니다.

### 장점 4. 회귀 테스트에 적합하다
모델 버전이 바뀌었을 때 지시 준수력이 올라갔는지, 내려갔는지 보기 좋습니다.

---

## 17. 이 저장소의 한계

이제 냉정하게 보겠습니다.

### 한계 1. “좋은 답변”을 평가하지 않는다
이건 의도된 한계입니다.  
사실성, 유용성, 논리성, 창의성은 거의 보지 않습니다.

### 한계 2. 검증 가능한 지시만 잘 본다
“친절하게 설명하라”, “감동적으로 써라”, “더 설득력 있게 써라” 같은 건  
이 저장소의 주특기가 아닙니다.

### 한계 3. 문자열/토크나이징 정의에 민감하다
문단, 단어, 문장 경계는 구현 선택의 영향을 받습니다.

### 한계 4. prompt exact match 방식이 다소 깨지기 쉽다
벤치마크 자체는 간단하지만, 파이프라인 연결에서는 꽤 예민합니다.

---

## 18. 누구에게 특히 유용한가

이 저장소는 아래 사람들에게 특히 유용합니다.

### 모델 평가 연구자
LLM judge 없이 instruction-following을 보고 싶을 때 좋습니다.

### 모델 서빙/플랫폼 엔지니어
출력 포맷 안정성을 회귀 테스트하고 싶을 때 매우 유용합니다.

### 프롬프트 엔지니어
모델이 실제로 제약을 지키는지 정량적으로 보고 싶을 때 좋습니다.

### 제품팀
“우리 모델이 점점 말을 안 듣는다”를 감이 아니라 수치로 확인할 수 있습니다.

---

## 19. 새 규칙을 추가하려면 어떻게 생각하면 되나

이 구조는 확장하기 꽤 쉽습니다.  
새 instruction을 추가할 때는 보통 이렇게 갑니다.

1. `instructions.py`에 checker class 추가
2. `build_description`, `check_following` 구현
3. `instructions_registry.py`에 id 등록
4. 필요한 conflict 규칙 추가
5. 테스트 케이스 추가
6. 데이터셋에 해당 규칙을 쓰는 예시 프롬프트 추가

즉, 이 저장소는 **새 규칙을 꽂아 넣기 좋은 구조**입니다.  
거대한 프레임워크는 아니지만, 그래서 오히려 손대기 쉽습니다.

## 20. 개인적인 총평

이 저장소의 미덕은 딱 세 가지입니다.

- 과장하지 않는다
- 범위를 좁힌다
- 채점을 기계적으로 끝낸다

이런 코드는 논문 발표용으로는 소박해 보여도,  
실전에서는 오래 갑니다.

왜냐하면 현장에서 필요한 건 “모델이 얼마나 똑똑하냐”보다  
**“시킨 포맷을 얼마나 안정적으로 지키냐”**인 경우가 정말 많기 때문입니다.

제가 이 저장소를 한 문장으로 평하면 이렇습니다.

> **LLM의 지시 준수력을, 말빨이 아니라 규칙으로 재단하는 깔끔한 평가기.**

## 22. 마지막으로, 아주 짧은 실무 조언

이 저장소를 그대로 가져다 쓸 생각이라면 아래 세 가지만 먼저 확인하십시오.

- 우리 파이프라인의 프롬프트 직렬화가 stable한가
- loose 점수를 공식 점수처럼 오해하지 않을 것
- paragraph / sentence / word 정의를 우리 제품 기대치와 맞출 것

이 세 가지만 챙겨도 절반은 먹고 들어갑니다.
